## 📋 迁移过程回顾

### 第 1 轮：通知功能分析（❌ 失败）

**问题**：总结旧版通知逻辑时出现大量错误假设

| 我的假设 | 实际代码 | 偏差原因 |
|---------|---------|---------|
| "每 10% 推送一次" | "每 5 分钟 OR subtask 变化" | 读了 `research.js` 但实际逻辑在 `research-worker.js` |
| "追问问题来自服务端" | "本地生成" | 没追踪调用链 |
| "完成通知有多个卡片" | "单个卡片" | 碎片化阅读 |

**根本原因**：
1. ❌ 没有 `grep` 找调用点
2. ❌ 没有追踪完整数据流
3. ❌ 基于"常识"推断，不是代码事实

**改进方法**：
```bash
# 1. 先找调用点
grep -rn "sendProgressNotification" /path/to/code/

# 2. 追踪数据流
read research-worker.js → notifier/index.js → formatProgress()

# 3. 贴代码确认
"我理解的是 XXX，对吗？"
```

**结果**：理解正确，实现顺利

### 第 3 轮：Prompt 优化实现（⚠️ 遇到技术陷阱）

**问题**：正则表达式不工作

```javascript
// ❌ 错误：正则中有空格
/(对比 | 比较 | 区别)/.test(topicLower)  // 永远返回 false

// ✅ 正确：用 includes 或移除空格
topicLower.includes('对比') || topicLower.includes('比较')
```

**根本原因**：
- JavaScript 正则表达式中的空格是**字面匹配空格**
- 中文字符在正则中需要确保编码正确
- 用 `includes()` 比正则更可靠（对于简单字符串匹配）

## 🎯 新教训总结

### 教训 1：代码分析"三步法"

**❌ 错误做法**：
1. 打开一个文件
2. 看到什么总结什么
3. 基于片段推断整体

**✅ 正确做法**：
```
Step 1: grep 找调用点
  → grep -rn "functionName" src/
  
Step 2: 追踪完整链路
  → 谁调用？→ 何时调用？→ 调用什么？→ 怎么响应？
  
Step 3: 贴代码确认
  → "我理解的是 XXX，对吗？"
```

**检查清单**：
- [ ] 我找到实际调用的地方了吗？
- [ ] 我追踪了完整数据流吗？
- [ ] 我的结论有代码原文支撑吗？
- [ ] 我跟用户确认理解了吗？

---

### 教训 2：技术实现"三验证"

**❌ 错误做法**：
1. 写代码
2. 假设能工作
3. 遇到问题再调试

**✅ 正确做法**：
```
验证 1: 语法检查
  → node -c src/file.js
  
验证 2: 单元测试
  → 写小型测试脚本验证核心逻辑
  
验证 3: 集成测试
  → 完整流程测试
```

**具体案例**：
```javascript
// 写完后立即测试
node -e "
const { detectQuestionType } = require('./src/utils/promptHelpers.js');
console.log(detectQuestionType('宁德时代和比亚迪对比'));
// 预期：comparative
// 实际：analytical → 发现问题，立即修复
"
```

### 教训 3：正则表达式陷阱

**JavaScript 正则的特殊性**：

| 写法 | 含义 | 问题 |
|------|------|------|
| `/(对比 | 比较)/` | 匹配"对比"或"比较" | ✅ 正确 |
| `/(对比 | 比较)/` | 匹配"对比 "或" 比较"（带空格） | ❌ 错误 |
| `/和.*比/` | 匹配"和...比" | ✅ 正确 |
| `/和.*比/` | 匹配"和.*比"（字面） | ❌ 错误 |

**最佳实践**：
```javascript
// 简单字符串匹配 → 用 includes()
if (topic.includes('对比') || topic.includes('比较'))

// 复杂模式匹配 → 用正则，但要测试
if (/(和.*比 | 与.*比)/.test(topic))

// 测试正则
console.log(/(对比)/.test('对比'));  // true
console.log(/(对比)/.test(' 对比')); // false (有空格)
```

### 教训 4：迁移策略"三不原则"

**❌ 不要**：
1. **不要完全照搬** - 旧版 2500 行，新版只需 350 行（14%）
2. **不要过度优化** - 先恢复 90% 功能，后续再增强
3. **不要忽视架构差异** - 旧版 CLI vs 新版 Module

**✅ 要**：
1. **要保留核心价值** - 意图分析、动态框架、实体提取
2. **要简化实现** - 用 `includes()` 替代复杂正则
3. **要符合新架构** - 纯函数、无状态、易测试

### 教训 5：文档驱动开发

**❌ 错误做法**：
- 写完代码再写文档
- 文档是"事后补充"

**✅ 正确做法**：
```
Step 1: 先写对比分析文档
  → CUE-RESEARCH-PROMPT-ANALYSIS.md
  
Step 2: 再写实现方案
  → 方案 A/B/C对比
  
Step 3: 实现后写总结
  → CUE-RESEARCH-PHASE1-COMPLETE.md
```

**好处**：
- 思路清晰
- 便于 review
- 后续维护有依据

## 💡 核心心法

> **"慢就是快"**
> 
> 花 30 分钟 `grep` + 追踪调用链，胜过 3 小时写错代码再返工。

> **"证据胜过假设"**
> 
> 贴 10 行代码原文，胜过 100 行"我觉得"。

> **"测试驱动信心"**
> 
> 写完立即测试，发现问题成本最低。
