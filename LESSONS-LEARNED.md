# Lessons Learned - 代码分析教训

## 2026-03-12: Cue Research 通知逻辑分析

### 问题
在分析旧版 cuebot 通知逻辑时，做出了大量错误假设：
- 假设"每 10% 推送一次" → 实际是"每 5 分钟 OR subtask 变化"
- 假设"追问问题来自服务端" → 实际是"本地生成"
- 假设"完成通知有多个卡片" → 实际是"单个卡片"

### 根源
1. **没有读对文件** - 读了 research.js 但实际逻辑在 research-worker.js
2. **没有追踪调用链** - 只看函数定义，没找谁调用它
3. **假设太多，验证太少** - 基于"常识"推断，不是代码事实
4. **碎片化阅读** - 没有把文件联系起来理解完整流程

### 正确方法
1. **先找调用点，再读实现**
   ```bash
   grep -rn "sendProgressNotification" /path/to/code/
   ```

2. **追踪完整数据流**
   ```
   谁触发？ → 何时触发？ → 触发什么？ → 发送什么？ → 怎么发送？
   ```

3. **少假设，多贴代码**
   - ❌ "应该是 XXX"
   - ✅ 贴代码原文 + "我理解的对吗？"

4. **先确认，再实现**
   - 在给出方案前，先问"我理解的是 XXX，对吗？"

5. **关注关键变量**
   - 找到决策变量（如 `elapsedMinutes`, `shouldNotify`）就理解了逻辑

### 核心教训
> **不要基于"常识"推断代码逻辑，直接读代码 + 追踪调用链。**

---

## 应用检查清单

下次分析代码时，先问自己：
- [ ] 我找到实际调用的地方了吗？
- [ ] 我追踪了完整数据流吗？
- [ ] 我的结论有代码原文支撑吗？
- [ ] 我跟用户确认理解了吗？
- [ ] 我关注了关键决策变量吗？
